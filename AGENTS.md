# AGENTS.md — правила для AI-агентов и контрибьюторов

Этот файл — точка входа для любого AI-агента (Cursor, Claude, Codex и т.п.),
работающего над этим проектом. Перед любой задачей агент **обязан** прочитать:

1. `AGENTS.md` (этот файл) — общие правила работы.
2. `project_description.md` — что и зачем мы строим.
3. `backlog.md` — текущая задача, её acceptance criteria и контекст.
4. Все файлы, перечисленные в поле **Component** конкретной задачи.

`CLAUDE.md` — это алиас на этот файл; правила одни и те же.

---

## 0. Core Principle

Изменение считается завершённым, только если:

- код проходит `make check` (форматирование, линтер, типы, тесты);
- `backlog.md` актуализирован (статус, чек-листы, блок **Resolution**).

---

## 1. Code Checks

### 1.1 Канонические команды

Единственный источник правды по проверкам — `Makefile`.

- `make check` — полный прогон: format check + lint + types + tests.
- `make format` — авто-форматирование (`black`, `isort`).
- `make lint` — `flake8`.
- `make types` — `mypy` (мягкий режим, см. `pyproject.toml`).
- `make test` — `pytest`.
- `make migrate` — применить миграции Alembic локально.

Конкретные тулзы и их флаги описаны в `Makefile` / `pyproject.toml` / `requirements-dev.txt`,
а не здесь.

### 1.2 Правила

- Новая логика → как минимум один тест.
- Если `make check` не зелёный — работа не закончена.
- Исключения линтера допускаются только инлайн с комментарием `# why: ...`.
- Одна задача = одна логическая единица изменения.
- Если правка задевает > 3 файлов — остановиться и уточнить с пользователем.
- Любой новый файл → добавить в **Technical Details** соответствующей задачи в `backlog.md`.

---

## 2. project_description.md — продуктовый контекст

См. `project_description.md`. Этот файл — единственный источник правды о том,
**что и зачем мы строим**. Агент читает его перед каждой нетривиальной задачей,
чтобы принимать решения в продуктовом контексте, а не вслепую.

Обновлять `project_description.md` нужно, когда меняется:

- продуктовая логика;
- стек или ключевые зависимости;
- архитектурные принципы;
- список анти-целей (что мы НЕ делаем).

---

## 3. backlog.md — единственный список задач

- `backlog.md` — единственный канонический список задач.
- У каждой задачи уникальный ID вида `TYPE-NNN`.
- Cursor / агент читает `backlog.md` без уточняющих вопросов и работает по ID.

Структура и обязательные поля задач описаны в `backlog.md` в начале файла
и должны соответствовать секциям 4–7 ниже.

---

## 4. Структура backlog.md

### 4.1 Секции по приоритету

- 🔴 High Priority
- 🟡 Medium Priority
- 🟢 Low Priority
- 🧊 Icebox (опционально)

### 4.2 Заголовок задачи

`### [TYPE-NNN] Title`

`TYPE` ∈ `FEATURE` | `BUG` | `TECH` | `UX` | `DOC` | `OPS`
`NNN` — трёхзначный номер.

---

## 5. Обязательные поля задачи

Каждая задача обязательно содержит, в этом порядке:

- **Status:**
- **Priority:** High | Medium | Low
- **Component:**

Допустимые статусы:

- 🆕 To Do
- 🟡 In Progress
- ⛔ Blocked
- 🧪 In Review
- ✅ Done

---

## 6. Обязательные блоки контента

Минимум для любой задачи:

1. **Problem Description**
2. **Expected Behavior**
3. **Technical Details**
4. **Acceptance Criteria** (чек-лист `- [ ]`)

Для `✅ Done` дополнительно обязательно:

5. **Resolution** — что сделано, где, ключевые решения.

---

## 7. Машиночитаемость (строго)

- Один ID — одна задача.
- Чек-листы только в формате `- [ ]` / `- [x]`.
- Пути к файлам всегда в обратных кавычках.
- Никакого свободного текста вместо статуса или чек-листа.
- Никаких дублей: одна и та же задача не может существовать под разными ID.

---

## 8. Обновление backlog.md

### При старте задачи

- Найти задачу в `backlog.md`.
- Перевести статус в `🟡 In Progress`.

### При частичном прогрессе

- Обновить чек-листы.
- Добавить блоки **Progress** и **Next**.

### При блокере

- Статус → `⛔ Blocked`.
- Добавить причину и условие разблокировки.

### При завершении

- Статус → `✅ Done`.
- Все пункты acceptance criteria отмечены `[x]`.
- Добавлен блок **Resolution**.

## 9. Промпты

- Все LLM-промпты живут в `prompts/` как `.txt` файлы.
- Имя файла: `{TYPE-NNN}_{short_name}.txt`.
- В шапке каждого файла: `model`, `temperature`, `purpose`, `version`.
- В `src/services/openai_service.py` промпты грузятся из `prompts/`,
  а не хардкодятся в коде (см. задачу `TECH-001`).

## 10. Внешние инструменты

### Общее правило

Для любого действия на внешней платформе — **сначала пробовать через MCP**.
Спрашивать пользователя руками только если MCP недоступен или действие упало.

### Render

Любые действия на Render (deploy, restart, env vars, логи и т.п.):

1. Сначала через **Render MCP**.
2. Если MCP не подключён или действие упало — остановиться и попросить пользователя сделать руками.
3. Никогда не просить пользователя сделать что-то на Render, не попытавшись через MCP.

### Telegram

- Не дёргать прод-бота для тестов; для тестов использовать отдельный токен и `/test` команду.
- Любые миграции БД сначала катать на Render-staging БД, если она появится.

## 11. Поведение агента

### Перед стартом задачи — обязательно прочитать:

- `project_description.md` — продуктовый контекст.
- `backlog.md` — найти задачу и acceptance criteria.
- Все файлы из поля **Component** задачи.

### Во время работы

- Следовать **только** acceptance criteria, не выдумывать новые требования.
- Если требования неполные или противоречат друг другу — **СТОП**, задать
  **один** уточняющий вопрос пользователю.
- Если пользователь прислал лог, ошибку или вывод бота — **не писать код сразу**:
  1. Объяснить проблему как ты её понял.
  2. Сказать что планируешь поменять и почему.
  3. Дождаться подтверждения перед тем как трогать код.

### После работы

- Обновить `backlog.md` (статус, чек-листы, **Resolution**).
- Прогнать `make check` локально — должен быть зелёным.
- Только после этого предлагать коммит / PR.

## 12. Стиль кода

- Python 3.11+ (см. `render.yaml` → `PYTHON_VERSION`).
- Форматирование: `black` (line length по умолчанию).
- Импорты: `isort` (профиль `black`).
- Типы: аннотации обязательны для публичных функций сервисов и хэндлеров.
- Docstring — для публичных функций и классов; внутренние комментарии
  только тогда, когда они объясняют **почему**, а не **что**.
- Никаких `print()` в проде — только `logging`.
- В роутерах нет бизнес-логики: они вызывают сервисы и форматируют ответ.
